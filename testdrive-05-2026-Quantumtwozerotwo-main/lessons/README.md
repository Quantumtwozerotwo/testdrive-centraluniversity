# Lessons

Здесь лежат задания для работы в парах на уроке.

Пара выбирает уровень и открывает готовую папку:

- `easy_balanced_brackets_task/`
- `medium_merge_intervals_task/`
- `hard_grid_dfs_task/`

В каждой папке уже есть:
- `solution.py` — сюда пишется код;
- `test_solution.py` — сюда добавляются тесты;
- `reference_tests.py` — итоговая проверка, её запускают в конце;
- `README.md` — короткая инструкция для выбранной задачи.

## Роли

Есть две роли:
- `driver` печатает код и тесты;
- `navigator` выбирает следующий тест-кейс и следит, чтобы пара не писала всё решение сразу.

В начале:
1. Решите, кто первый `driver`, а кто `navigator`.
2. Откройте папку задачи.
3. Запустите `pytest -q`.
4. Убедитесь, что первый тест красный.

## Один цикл

### 1. Красный тест

`navigator` выбирает следующий тест-кейс из списка в задании.

`driver` пишет ровно один тест в `test_solution.py`.

Пара запускает:

```bash
pytest -q
```

Тест должен упасть. Если он не упал, тест не проверяет новое поведение.

### 2. Зелёный тест

`driver` пишет минимальный код в `solution.py`.

Цель — пройти текущий тест, а не решить всю задачу целиком.

Пара снова запускает:

```bash
pytest -q
```

### 3. Короткая уборка

Если код стал неудобным, пара тратит 1-2 минуты на правку.

После правки тесты снова должны быть зелёными.

### 4. Смена ролей

Когда тест зелёный, роли меняются:
- бывший `driver` становится `navigator`;
- бывший `navigator` становится `driver`.

Потом начинается следующий цикл.

## Пример одного шага

Это не часть заданий, просто пример порядка действий.

Допустим, есть функция:

```python
def double(x):
    pass
```

`navigator` предлагает простой тест-кейс: `double(2)` должен вернуть `4`.

`driver` пишет тест:

```python
def test_double_two():
    assert double(2) == 4
```

Пара запускает `pytest -q` и видит красный тест.

`driver` пишет минимальный код:

```python
def double(x):
    return 4
```

Пара снова запускает `pytest -q`. Тест зелёный.

Теперь роли меняются. Следующий `navigator` выбирает новый тест-кейс, который заставит улучшить код.

## Как выбрать уровень

- `easy_balanced_brackets.md` — разминка на 15-20 минут. Берите, если хотите спокойно освоить TDD.
- `medium_merge_intervals.md` — обычная задача на 25-35 минут. Берите, если уже понятно, как писать тесты.
- `hard_grid_dfs.md` — длинная задача на 35-50 минут. Берите, если готовы к обходу графа.

## Общие правила

- Пишите решение на Python.
- Запускайте тесты через `pytest`.
- Не пытайтесь написать всё решение сразу.
- Один шаг — один тест.
- Роли меняются только после зелёного теста.
- Если застряли, вернитесь к более простому тесту.
- Если спорите о решении дольше двух минут, напишите более простой тест.

## Финальная проверка

Когда пара считает задачу готовой, запустите эталонные тесты:

```bash
pytest -q reference_tests.py
```

Эти тесты не нужны для каждого шага. Они нужны только в конце, чтобы понять, что задача закрыта.
